## 1. 핵심 원칙: Rocket with Tight Bolts 🚀🔩

> "가설은 창의적으로 사유하되, 근거 데이터와 판단은 엄밀한 팩트에 근거한다."

PLEDS는 하이 리스크 포트폴리오를 운용한다. 로켓이 가는 길에 사소한 나사 풀림이 있어서는 안 된다.

### 이중 사고 체계 (Dual Thinking Framework)

| | 🎨 Alpha Thinking (창의적 사유) | 🔬 Fact Grounding (팩트 기반) |
|---|---|---|
| **영역** | 가설 생성, 테시스 구축, 시나리오 상상 | 데이터 검증, 수치 판단, 리스크 계량 |
| **허용** | 대담한 아날로지, 비선형 연결, 역발상, 비컨센서스 | 검증된 수치만, 출처 필수, 교차검증 |
| **금지** | — | 추정치를 팩트처럼 사용, 출처 없는 수치, 소스 불일치 무시 |
| **예시** | "BMNR은 2022 MSTR의 재현이다" ✅ | "ETH NAV $10.5B" → SEC filing + CoinGecko 교차확인 필수 ✅ |
| **적용** | Phase 1-5 전문가 토론의 **논리/가설** 부분 | Phase 0/0.5 데이터 + 전문가 토론의 **수치 인용** 부분 |

### 적용 규칙
1. **가설(Thesis)과 근거(Evidence)를 명시적으로 분리 표기**
   - 가설: `[THESIS]` 태그 — 창의적 사유, 검증 전 아이디어
   - 근거: `[FACT]` 태그 — 검증된 데이터에 기반한 판단
2. **전문가는 가설을 자유롭게 제시하되, 수치를 인용할 때는 반드시 Phase 0 검증 데이터만 사용**
3. **"느낌"으로 숫자를 만들지 않는다** — 데이터가 없으면 `[데이터 부재]`로 명시하고, 그 상태에서 판단
4. **Alpha는 팩트 위에서만 구축한다** — 단단한 현실 인식 → 자유로운 알파 구상

## 2. AI Commoditization Filter 🤖

> "AI가 연역적 처리를 commoditize한 시대, 분석 능력은 해자가 아니다.
> 진짜 병목은 상류에 있다 — 독점적 데이터 원천과 관점의 생산."

### 원칙
모든 기업의 해자를 평가할 때 아래 질문을 필수로 적용한다:

**질문 1: "이 기업의 핵심 가치가 AI로 대체/commoditize되는가?"**

| 해자 유형 | AI 시대 내구성 | 예시 |
|----------|--------------|------|
| 🔴 분석/처리 능력 | **약화** — AI가 더 잘함 | 컨설팅, 리서치, 소프트웨어 분석 |
| 🟡 소프트웨어/알고리즘 | **불확실** — AI가 빠르게 추격 | SaaS, 분석 도구 |
| 🟢 독점 데이터 원천 (Proprietary Data) | **강화** — AI가 데이터를 더 잘 쓸수록 원천 가치 상승 | 위성 영상, 유전자 DB, 센서 네트워크 |
| 🟢 물리적 인프라/센서 | **최강** — AI가 물리를 대체 못함 | PX(ERII), 위성(SATL/BKSY), 시퀀서(ILMN), 발사체(RKLB) |
| 🟢 규제/인증 해자 | **강화** — AI가 FDA/ITAR 인증을 우회 못함 | 의약품, 방산, 원자력 |
| 🟢 네트워크 효과 + 데이터 축적 | **강화** — 쓸수록 데이터 쌓여 해자 확대 | USDC(CRCL), 거래소 데이터 |

**질문 2 (Search Problem Lens): "이 기업의 핵심 문제가 compute + search로 풀려버리는가?"**

> 출처: AI Frontier EP91 (노정석/최승준, 2026-03-21) — "모든 문제는 compute을 이용한 search problem으로 수렴한다"

AI 시대의 문제 해결 = **명확한 목표 + evaluation metric + compute 투입 → 최적화**. 이 구조로 치환 가능한 기업의 가치는 급격히 하락한다.

| Search 치환성 | 해자 전망 | 예시 |
|-------------|----------|------|
| 🔴 **쉽게 치환** — 목표+평가가 디지털 내에서 완결 | 해자 소멸 | 분석 서비스, 코드 생성, 번역, 요약 |
| 🟡 **부분 치환** — 일부 영역만 search로 풀림 | 해자 축소 | SaaS (기능은 풀리지만 데이터/워크플로우 잔존) |
| 🟢 **치환 불가** — 물리 세계의 보상 신호 없이는 search 불가능 | 해자 강화 | 아래 "Verification Moat" 참조 |

**질문 3 (Verification Moat): "이 기업만이 생성할 수 있는 보상 신호(verification)가 있는가?"**

> 핵심 원리: RLVR(Reinforcement Learning by Verifiable Rewards)이 모든 도메인으로 확장 중.
> **디지털에서 검증 불가능한 보상 신호를 물리적으로 생성할 수 있는 환경 = 궁극의 해자.**

| Verification 유형 | 해자 등급 | 예시 |
|------------------|----------|------|
| 🔴 **디지털 완결 검증** — AI가 자체적으로 맞/틀 판단 가능 | **No Moat** | 코드(컴파일), 수학(증명), 텍스트 품질 |
| 🟡 **규제/제도 검증** — 인간 기관의 승인이 필요 | **Partial Moat** | FDA 승인, 금융 라이선스, ITAR 인증 |
| 🟢 **Atom World 검증** — 물리적 실험/센서/인프라에서만 보상 신호 발생 | **Deep Moat** | 위성 촬영(SATL), 에너지 회수(ERII), 재료 실험(Periodic Labs) |
| 🟢 **자산 규모 검증** — 대규모 자산의 보유·운용 자체가 검증 불가능한 능력 | **Scale Moat** | ETH $10B Treasury(BMNR), 결제 네트워크(CRCL), 거래소 유동성(COIN) |

**Asset Retention = Verification Moat인 이유:**
- $10B 규모의 ETH를 축적하려면: 자본조달 능력 + 기관 신뢰 + 규제 통과 + 시장 타이밍
- 이 과정은 AI가 compute으로 search할 수 없음 — **인간 세계의 신뢰와 자본이 보상 신호**
- 규모 자체가 네트워크 효과 생성: mNAV 프리미엄, 스테이킹 수익률, ETH supply 점유율
- "자산 보유 + 유지(retain)" = 시간이 지날수록 복제 불가능성 증가

### 적용 규칙
1. **L3(Value Chain) + L4(Company) 분석 시 3개 질문 모두 필수 적용**
2. "AI 분석 능력"을 핵심 해자로 주장하는 기업은 **자동 1단계 할인**
3. "독점 데이터 원천 + 물리적 수집 인프라"를 소유한 기업은 **Constraint Premium 가산**
4. **Atom World / Scale Moat 보유 기업은 추가 Premium 가산**
5. 평가 시 구분: **데이터 축적형**(내구적, 시간이 해자) vs **데이터 실시간형**(소비되면 끝)
6. 전문가 토론에서 모든 Bull case에 "이 해자가 AI search로 풀려버리지 않는가?" 반론 의무
7. **Search Problem Lens는 §7 Conviction Card의 C1(태울 가치?) 판단에 직접 반영** — 치환 불가 해자 보유 시 conviction 상향 근거

### 비판적 수용 원칙
이 프레임워크 자체도 맹신하지 않는다:
- 데이터 원천이 해자라도, **수요가 없으면 무의미** (데이터 자체는 비용)
- "관점의 생산"은 넘쳐남 — 진짜 희소한 건 **검증된 1차 데이터**
- 분석 commoditize ≠ 분석 무가치. **독점 데이터 + AI 분석 결합**이 최강
- Search Problem Lens도 만능이 아님 — **문제 정의(objective) 자체가 모호한 영역**에서는 search가 작동하지 않음. 취향, 의지, 비전 = 여전히 인간 영역

### 바닐라 원칙 🍦

> 출처: AI Frontier EP91 — "돌고 돌아 바닐라가 답이다. 데이터 커넥터 + 프롬프트 + 프론티어 모델 = 최강."

**PLEDS 시스템 자체의 진화 원칙:**

1. **모듈을 늘리지 않는다** — 새 인사이트가 생기면 기존 모듈을 업그레이드하지, 새 모듈을 만들지 않는다. 복잡도는 적이다.
2. **데이터 커넥터 품질 > 모듈 수** — API 연동(FRED, Twelve Data, CoinGecko, SEC EDGAR)의 안정성과 신선도가 전문가 수보다 중요하다.
3. **프롬프트 정제 > 구조 변경** — METHODOLOGY.md의 표현을 갈고닦는 것이 Phase를 추가하는 것보다 PLEDS 성능을 더 올린다.
4. **프론티어 모델의 capability overhang 활용** — 모델이 강해지면 전문가 페르소나의 일부가 불필요해진다. 분기 1회 prune 검토.
5. **도메인 이해 > 엔지니어링** — 투자 도메인에 대한 깊은 이해가 시스템 복잡도보다 결과를 좌우한다.

**Prune 기준 (분기 1회):**
- 최근 3개월 분석에서 해당 모듈이 판정을 바꾼 적이 있는가?
- 없으면 → 단순화 또는 다른 모듈에 흡수 검토
- 있으면 → 유지

## 3. Reality Grounding Layer 🌍

> "PLEDS가 세상과 연결되어 있는가?"
> 데이터 인프라의 건강, 정보 원천의 신뢰, 시스템의 현실 인식 정합성을 보장한다.

### 3-1. Data Source Health Check (데이터 소스 활력 점검)

> 모든 데이터 소스의 최신성과 신뢰도를 정기 점검한다.
> **상세 절차·소스 목록·장애 대응은 `DATA-SOURCES.md`에 단일 관리.**
> 소스 장애 시 `[DATA SOURCE ALERT]` 발행 + 대체 소스 전환 + Shawn 알림.

---

### 3-2. User Input Validation (사용자 입력 교차검증)

> "Shawn의 말도 팩트가 아니다. 검증한다."

Shawn이 제공하는 데이터(진입 가격, 수량, 날짜 등)도 **가용 데이터와 교차검증**한다:
- **진입 가격** → 해당 종목의 거래 범위(52주, 역대)와 대조. 범위 밖이면 즉시 되물음.
- **날짜** → 거래일 여부 확인
- **수치** → 자릿수/단위 상식 체크 (ERII $95 → 역대 최고 ~$27, 불가능 → "오기 아닌지?" 확인)

**불일치 발견 시**: 즉시 되물음. 무조건 수용 금지.
**원칙**: Shawn을 존중하되 맹종하지 않는다. 오기/착각을 잡아주는 것이 진짜 신뢰.
**교훈**: 2026-03-21 ERII "$95" 사건 — 역대 최고가 $27인 종목에 $95 진입 수용. 1초면 잡을 수 있는 오기를 무조건 수용한 실패.

---

### 3-3. Information Source Expansion (정보 원천 확충)

현재 데이터 소스 너머, 세상에 대한 **깊이 있는 이해**를 위한 고신뢰 정보 소스 체계:

#### Tier 1 (최고 신뢰 — 1차 데이터)
- SEC EDGAR filings (10-K, 10-Q, 8-K, DEF 14A proxy)
- FRED / BLS / Census (정부 통계)
- 기업 IR 페이지 (earnings transcript, investor presentation)
- Patent filings (USPTO, Google Patents)

#### Tier 2 (높은 신뢰 — 전문 분석)
- 학술 논문 (Google Scholar, arXiv, SSRN)
- 산업 보고서 (IEA, WHO, FAO 등 국제기구)
- 중앙은행 발간물 (Fed, ECB, BOJ working papers)
- 전문 미디어 (Reuters, Bloomberg 발췌)

#### Tier 3 (참고 — 2차 해석)
- 투자 리서치 (Seeking Alpha, Motley Fool — 편향 인지 필수)
- 유튜브/팟캐스트 (전문가 인터뷰 — 발언자 신원 확인)
- 소셜 미디어 (X/Reddit — 센티먼트 지표로만, 팩트로 사용 금지)

#### Tier 4 (경계 — 노이즈)
- 익명 게시판, 루머, 출처 불명 텔레그램 채널
- **→ PLEDS 입력 금지. 참고 시 `[Tier 4 — 미검증]` 명시**

**규칙:**
- 전문가 토론에서 인용 시 **Tier 표기 의무** (예: `[Tier 1: SEC 10-K]`)
- **Tier 1-2만 `[FACT]` 태그 허용**
- Tier 3은 `[THESIS]` 또는 `[REFERENCE]`로만
- 새로운 고품질 소스 발견 시 `DATA-SOURCES.md`에 등록

### 3-4. Reality Coherence Audit (현실 인식 정합성 감사)

PLEDS 시스템 자체의 현실 인식이 왜곡되지 않았는지 **메타 레벨에서 점검**:

| # | 점검 항목 | 내용 | 주기 |
|---|----------|------|------|
| 1 | **예측 vs 실제 괴리** | 지난 30일 PLEDS 판정과 실제 시장 결과 비교. 방향 적중률, 크기 오차 | 주간 |
| 2 | **Stale Thesis Detection** | 30일 이상 업데이트 없는 conviction → 자동 재검토 플래그 | 주간 |
| 3 | **Echo Chamber Check** | 최근 5건의 분석에서 Bull/Bear 비율이 80/20+ 치우침 → 경고 | 주간 |
| 4 | **Model Drift** | 밸류에이션 모델의 전제(금리, 성장률 등)가 현실과 괴리 감지 | 주간 |
| 5 | **Blind Spot Scan** | "우리가 보지 못하고 있는 리스크는?" — 의무 실행 | 월간 |

**실행:**
- **주간 리뷰(일요일)**: 항목 1-4 자동 점검
- **월간 리뷰**: 항목 5 (Blind Spot Scan) 실행
- **적중률 60% 미만**: 시스템 가중치/절차 재검토 트리거

**산출물:** `audit/YYYY-MM-reality-audit.md`

### 3-5. Regime Sentinel (선제적 레짐 감지) 🚦

> "반응형 레짐 판정(VIX 25 돌파 후)을 보완하여, 레짐 전환을 선제적으로 감지한다."

PLEDS의 Macro Layer가 레짐을 판정하지만, 이는 **반응형**이다. Regime Sentinel은 5개 핵심 지표의 임계치 조합으로 **선제적 감지**를 제공한다.

#### 모니터링 지표

| 지표 | 소스 | 임계치 | 의미 |
|------|------|--------|------|
| VIX | Yahoo ^VIX | >25 | 공포 시작 |
| 2-10 Spread | FRED T10Y2Y | <0 (역전) | 경기 침체 신호 |
| HY Spread | FRED BAMLH0A0HYM2 | >400bp | 신용 리스크 급등 |
| BTC 주간 변동 | CoinGecko | <-15% | 크립토 레짐 전환 |
| DXY 월간 변동 | Twelve Data | >+3% | 달러 강세 레짐 |

#### 경보 단계

| 돌파 수 | 상태 | 액션 |
|---------|------|------|
| 0개 | 🟢 정상 | 상태 표시만 |
| 1개 | 🟡 주의 | 모니터링 강화, 해당 지표 강조 |
| 2개 | 🟠 경고 | Macro Layer 긴급 재판정, Shawn 알림 |
| 3개+ | 🔴 레짐전환 | 전 포지션 재검토, Shawn 즉시 알림 |

#### 규칙
- 일일 브리핑 Phase 0에서 `data/regime-sentinel.py` 자동 실행
- 🟠 경고 이상 시 텔레그램 즉시 알림 (일일 브리핑 전이라도)

## 4. Fresh Eyes Protocol 👁️

> "반복 평가가 진짜 독립적인가, 첫 번째 평가의 관성인가?"

재평가 시 이전 판정의 앵커링을 차단하기 위한 필수 절차.

### 규칙 1: Zero-Base Re-Rating (영점 리셋)
- 재평가 시작 시 **이전 확신도/적정가를 보지 않고** 시작
- "이 종목을 오늘 처음 본다. 현재 가격, 현재 팩트만으로 판단하라"
- 이전 보고서는 **독립 판정을 먼저 내린 후**에만 비교 허용

### 규칙 2: Mandatory Bear-First (반드시 곰부터)
- 모든 재평가는 **Bear case를 먼저** 구축
- "이 종목이 -50% 가는 시나리오는?" 부터 시작
- Bull case는 Bear를 반박하는 형태로만 제시
- Bear case가 설득력 있으면 확신도 자동 하향

### 규칙 3: Delta Accountability (변화 책임)
- 이전과 같은 판정이면: **"왜 같은지"** 새로운 근거를 명시적으로 설명
- 새로운 근거가 없고 "이전과 동일"이면 → **자동 반 단계 하향** (관성 페널티)
- 변경이면: Delta Journal (🟢New Fact / 🟡Model Refinement / 🔴Mood Shift) 기록

### 규칙 4: Event Force-Address (이벤트 강제 반영)
- 주요 이벤트(M&A 무산, 실적, 규제, 경쟁 등) 발생 시 **독립 섹션**으로 분석 의무
- "시장의 실수" vs "시장이 맞다" 양쪽을 **동등한 분량**으로 논증
- 어느 쪽이든 한쪽만 쓰면 불완전 판정

### 적용 시점
- 일일 브리핑의 보유 종목 리뷰
- Bottleneck Scan 재스캔
- 종목 재평가 요청 시
- Conviction Card 업데이트 시

## 5. Price Drop Triage Protocol 🩺

> "가격이 떨어졌을 때, 무조건 '싸졌다' 또는 '끝났다'로 점프하지 않는다.
> 3가지 가설을 동등한 분량으로 검증한다."

### 발동 조건
- **보유종목**: 고점 대비 -10% 이상 하락
- **워치리스트**: 52주 저점 근접 (저점 대비 +5% 이내)

### 3가지 가설 (동등 분량 검증 의무)

| A. 시장 실수 (Mr. Market) | B. 펀더멘탈 훼손 (Thesis Break) | C. 정보 비대칭 (Information Gap) |
|---------------------------|-------------------------------|-------------------------------|
| 센티먼트/공포 | TAM 축소 | 내부자 매도 패턴 |
| 섹터 연좌제 | 경쟁 구도 변화 | 공매도 급증 + 근거 |
| 매크로 오버슈트 | 마진 구조 붕괴 | 소송/규제 시그널 |
| 유동성 이벤트 | 경영진 이탈 | 채널체크 악화 루머 |
| **→ 매수 기회** | **→ Kill 검토** | **→ 추가 조사 후 판단** |

### 검증 절차
1. 각 가설에 **최소 2개 근거** 제시 의무
2. 근거가 없으면 `[근거 부재]`로 명시 — 해당 가설의 신뢰도 저하
3. 3개 가설의 확률을 명시적으로 산출 (합 100%)
4. **가장 높은 확률의 가설에 따라 행동**:
   - A 우세 → 추가 매수 검토
   - B 우세 → Kill Condition 재검토, 손절 준비
   - C 우세 → 추가 조사, 포지션 유지 but 경계

### 예시
```
BMNR -10.47% (주간)
A. 시장 실수 (60%): VIX 27 급등, 글로벌 리스크오프, 크립토 섹터 전체 하락
B. 펀더멘탈 훼손 (10%): ETH 네트워크 이슈 없음, 경영진 변화 없음
C. 정보 비대칭 (30%): 내부자 매도 확인 필요, 공매도 데이터 확인 필요
→ A 우세: 추가 매수 검토 구간
```

## 7. Conviction Card & Convergence Framework 🎯

> v4.0: C0 Thesis + Variant Perception Box + Edge Score 추가 (2026-03-26)
> "토론은 자유롭게 발산하되, 결론은 **세상의 변화에 대한 가설**로부터 시작하여 카드로 수렴한다."

PLEDS의 모든 산출물은 궁극적으로 아래 질문들에 답해야 한다. 단, 이 질문은 **전문가 토론(Phase 1-5) 중에는 의식하지 않는다.** 토론은 창의적으로 발산하고, **Allocator(Phase 6)가 토론 결과를 읽고 수렴한다.**

### Conviction Card (종목/테마당 1세트)

| Card | 질문 | 내용 |
|------|------|------|
| **C0 🔭 Thesis** | 세상이 어떻게 바뀌는가? | 이 기업이 존재하는 세상의 구조적 변화 가설. 가격 논의 전에 세상의 변화부터. |
| **C1 🔥 Burn?** | 컨빅션 있게 크게 태울 만한가? | 확신도 ⭐~⭐⭐⭐ + R:R 3:1 이상 + **1-2년 내 카탈리스트** + 핵심 근거 1줄 |
| **C2 🚪 Entry** | 진입 조건은? | 가격 레벨, 시점, 이벤트 트리거 (복수 조건 가능) |
| **C3 🎯 Exit** | 목표가/익절 조건은? | 시간축별(6mo/1yr/2yr) 목표가 + 조건부 익절 시나리오 |
| **C4 ☠️ Kill** | 가설 폐기/손절 조건은? | 가격, 이벤트, 기한 — 이 중 하나라도 충족 시 기계적 실행 |

### C0 Thesis — 세상의 변화 가설 (v4.0 신설) 🔭

> "이 기업이 존재하는 **세상**이 1-2년 후 어떻게 달라지는가? 그 변화로 3x가 가능한가?"

C0는 모든 Conviction Card의 **기초**. C1-C4는 C0 위에 세워진다.

**C0 필수 요소:**
1. **구조적 변화 가설** — "X가 Y로 바뀐다" (1문장)
2. **변화의 인과 메커니즘** — 왜 이 변화가 불가피한가? (물리적/경제적/인구학적 이유)
3. **변화의 시간축** — 이 변화가 언제 가시화되는가? **⚠️ 1-2년 내 가시화 여부 필수 판정**
4. **수혜 메커니즘** — 이 변화가 이 기업의 매출/이익에 어떻게 연결되는가?
5. **3x 잠재력** — 이 변화가 가시화될 때 현재가 대비 3x 상승 경로가 있는가? (v4.1 추가)
6. **반증 조건** — 이 변화가 일어나지 않는다면 어떤 신호가 먼저 나타나는가? **반증 기한: 최대 2년**

**Catalyst Proximity Filter (v4.1 신설):**
> "카탈리스트 없는 좋은 회사 = 시간 낭비. 시간 = 비용."

모든 종목에 **1-2년 내 가시적 카탈리스트** 식별 의무:
- 구체적 이벤트 (어닝, 규제, 제품 출시, M&A 등)
- 발생 예상 시점 (분기 단위)
- 카탈리스트 없으면 → **WAIT** (아무리 좋은 회사도)
- 실증: IREN(AI 채굴 전환), RKLB(발사 빈도 급증), LEU(HALEU 독점기), 삼성/하이닉스(HBM 사이클)

**C0 예시:**
```
C0 🔭 ERII — "2030년까지 전 세계 담수화 용량이 현재의 2.5배로 확대된다"
  인과: 인구 증가 + 기후변화 → 물 스트레스 인구 40억 → 담수화 유일한 해법
  시간축: 이미 진행 중 (MENA 메가플랜트 발주 2024-2028)
  수혜: PX 장비 독점 → 담수화 1GW당 ERII 매출 ~$30M
  반증: 2027년까지 담수화 신규 발주 연 $5B 미만 유지 시 가설 약화
```

**C0 → C1 연결 규칙:**
- C0의 세상 변화가 **높은 확률로 실현되고** + 이 기업이 **필수적 수혜자**이면 → C1 conviction 상향
- C0의 세상 변화가 **불확실**하거나 + 수혜 메커니즘이 **간접적**이면 → C1 conviction 하향
- **C0 없이 C1을 작성하면 불완전** — "왜 이 기업에 투자하는가?"의 근본 답이 빠짐

### Variant Perception Box (v4.0 신설) 📌

> "Edge = 시장과 다르게 보고, 내가 맞는 것"
> 컨센서스와의 차이를 명시적으로 추적하지 않으면 edge가 있는지조차 모른다.

**모든 PLEDS 분석에 필수 포함:**

```
📌 Variant Perception Box
━━━━━━━━━━━━━━━━━━━━━━
시장 컨센서스: [Sell-side PT 평균, 애널리스트 Buy/Hold/Sell 분포, 주요 내러티브]
PLEDS 뷰:     [우리의 판단 + 근거]
차이점:       [구체적으로 어디서 다른지]
차이의 근거:   [왜 시장이 틀렸다고 보는지 — 인과 메커니즘 기반]
검증 시점:     [이 차이가 드러나는 이벤트/데이터/시점]
Edge 판정:   [🟢 Strong / 🟡 Moderate / 🔴 No Edge]
━━━━━━━━━━━━━━━━━━━━━━
```

**Edge 판정 기준 (v4.2 단순화):**

Edge Test 4문항(E1-E4)의 통과 수 + 컨센서스와의 차이를 종합하여 **3단계**로 판정한다.
입력이 정성적 YES/NO이므로, 점수 공식 대신 판정 등급을 사용한다.

| Edge 등급 | 조건 | 투자 행동 |
|----------|------|----------|
| 🟢 **Strong Edge** | 4/4 YES, 또는 3/4 YES + 컨센서스 반대 | 적극 포지션 구축 (conviction sizing) |
| 🟡 **Moderate Edge** | 3/4 YES, 또는 2/4 YES + 컨센서스 반대 | 소규모 진입 또는 워치리스트 (확인 대기) |
| 🔴 **No Edge** | 2/4 이하 YES (컨센서스 반대 아닌 경우) | **패스** — edge 없는 곳에 돈을 넣지 않는다 |

**원칙:**
- 컨센서스와 방향이 반대일 때 한 단계 상향 허용
- E1-E4 각 문항은 정성적 판단이므로 "확실한 YES"와 "아마도 YES"를 구분하여 기록
- 경계선(2/4 + 컨센서스 반대 등)에서는 Moderator 재량으로 판정하되, 근거를 명시

**PT 연동:**
- Variant Perception Box를 judgment 파일에 기록
- Checkpoint에서 **"누가 맞았는가?"** 채점 — PLEDS vs 컨센서스
- `db/expert-accuracy.json`의 `variant_perception_accuracy` 필드에 누적
- **이것이 PLEDS의 진짜 KPI** — 단순 방향 적중률이 아닌, **컨센서스와 달랐을 때의 적중률**

### 적용 규칙

1. **Phase 1-5 (발산)**: 전문가들은 C0-C4를 의식하지 않음. Deep Loop + Cross-Comparison에 집중.
2. **Phase 6 (수렴)**: Allocator가 토론 결과를 읽고 **C0부터** 작성. C0 → C1 → C2 → C3 → C4 순서.
3. **Variant Perception Box**: Phase 6에서 Allocator가 작성. sell-side 컨센서스는 web_search로 수집.
4. **일일 브리핑**: C2/C3/C4 조건 달성 여부 + **VP Box 검증 시점 도래 여부** 체크.
5. **카드 없는 투자 제안은 불완전** — C0 또는 VP Box가 빠져도 불완전.
6. **Edge 🔴 No Edge이면 자동 PASS** — "좋은 회사인데 비싸다"로 끝나는 분석은 투자 대상이 아님.

> 에코 챔버가 아닌, 시장이 모르는 것을 찾아내는 시스템.

### 핵심 전환

| 이전 (v3.1) | 이후 (v4.0) |
|-------------|-------------|
| 4라운드 → 종료 | **Edge가 나올 때까지 Loop** |
| 같은 섹터 내 비교 | **이종 산업 창의적 비교** |
| 컨센서스 정리 | **컨센서스와 다른 곳을 찾는 것이 목표** |
| 절차 체크리스트 | **통찰의 품질이 종료 조건** |

### 8.1 Inner Loop: 4-Round Debate (기존 유지)

| Round | 이름 | 내용 |
|-------|------|------|
| **R1** | 독립 제시 | 각 전문가가 독립적으로 뷰 제시 |
| **R2** | Critic 반론 | Layer Critic이 구체적 반론 (의무) |
| **R3** | 재반박/수정 | defend / revise / abandon |
| **R4** | Moderator 평가 | **Edge Test 실시** (아래 10.2) |

**"살아남은 논거" 정의 (v4.1 명시화):**

R3 결과에 따라 논거의 상태가 결정된다:

| R3 결과 | 상태 | Moderator 처리 |
|---------|------|---------------|
| **defend 성공** | ✅ 살아남음 | R4에서 인용, Edge Test 대상 |
| **revise** | ✅ 살아남음 (수정됨) | 수정된 버전으로 R4 인용. 원본과 수정본 모두 기록 |
| **abandon** | ❌ 사망 | R4에서 제외. "폐기 이유" 1줄 기록 |
| **defend 실패** (Critic 반론 미반박) | ❌ 사망 | Moderator가 판정. 반론에 대한 재반박이 없거나 설득력 부족 시 |

**원칙:**
- defend와 revise는 모두 "살아남은" 것으로 간주
- abandon만 제외됨
- defend 실패 판정은 Moderator의 권한 (Critic 반론 > 원 주장일 때)
- **Moderator는 R4에서 "살아남은 논거 N개 / 사망 M개"를 명시적으로 집계**

### 8.2 Outer Loop: Edge Convergence Loop (신설) 🔄

> 적용 Layer: **L2 Sector, L3 Value Chain, L4 Company**
> L1 Macro, L5 Technical은 단일 Loop (데이터 기반이라 반복 토론 효과 제한적)

**Moderator가 R4에서 Edge Test 4문항을 평가:**

| # | Edge Test 질문 | YES 기준 |
|---|---------------|----------|
| **E1 Novelty** | 이 인사이트가 sell-side 컨센서스에 없는가? | web_search로 확인: 최근 90일 내 커버리지 있는 애널리스트 보고서/PT에서 언급되지 않은 관점. 커버리지 0인 소형주는 "시장 내러티브"(SeekingAlpha, 주요 뉴스) 기준 |
| **E2 Causality** | 명확한 인과 메커니즘이 있는가? (상관관계 아닌 인과) | "A이면 B이다"의 물리적/경제적 이유를 설명 가능 |
| **E3 Actionability** | 이 인사이트가 실제 투자 행동을 바꾸는가? | "이걸 몰랐다면 다른 결정을 했을 것" |
| **E4 Falsifiability** | 이 인사이트가 틀렸음이 증명될 수 있는가? | 구체적 반증 조건과 시점이 정의 가능 |

**Loop 규칙:**

```
loop_count = 0
MAX_LOOPS = 5

while loop_count < MAX_LOOPS:
    Run Inner Loop (R1→R2→R3→R4)
    
    Moderator runs Edge Test:
    if E1+E2+E3+E4 모두 YES:
        → "🎯 Edge Found" — Loop 종료, 인사이트 기록
    elif loop_count == MAX_LOOPS - 1:
        → "⚪ No Edge" — "시장이 효율적" 결론도 유효한 결론
    else:
        → Moderator가 가장 약한 E항목 식별
        → 해당 방향으로 심화 질문 생성 (Drill Question)
        → 다음 Loop에서 Drill Question을 R1 시드로 투입
    
    loop_count++
```

**Drill Question 예시:**

| 약한 항목 | Moderator가 생성하는 Drill Question |
|----------|----------------------------------|
| E1 Novelty 부족 | "sell-side가 이미 아는 것 말고, 이 밸류체인에서 **아무도 이야기하지 않는** 변수는?" |
| E2 Causality 약함 | "A→B 연결의 **물리적/제도적 메커니즘**을 한 단계 더 풀어라. 왜 A가 B를 야기하는가?" |
| E3 Actionability 부족 | "이 인사이트를 알면 **구체적으로 무엇을 다르게 하는가?** 포트폴리오 액션으로 번역하라." |
| E4 Falsifiability 부족 | "이 테시스가 **틀렸다면 6개월 내에 어떤 데이터가 나타날 것인가?**" |

### 8.3 Cross-Domain Creative Comparison (이종 비교 엔진) 🌐

> "자동차는 넷플릭스와 경쟁한다" — 같은 자원(시간, 돈, 관심)을 놓고
> 겉보기에 다른 산업이 실제로는 같은 수요 풀을 두고 경쟁한다.

**적용 Layer:** L2 Sector + L3 Value Chain + L4 Company

**비교 축 5가지 (Moderator가 최소 2개 탐색 의무):**

| # | 비교 축 | 질문 | 예시 |
|---|---------|------|------|
| **X1** | 수요 풀 경쟁 | 같은 고객의 같은 예산/시간/관심을 두고 경쟁하는 이종 산업은? | 자동차 vs 넷플릭스 (여가 시간), 대학교 vs 부트캠프 (교육 예산) |
| **X2** | 기능 대체 | 전혀 다른 기술이 같은 job-to-be-done을 수행하는 경우는? | 담수화(ERII) vs 클라우드 시딩 vs 재활용수 (물 확보), HALEU(LEU) vs SMR vs 지열 (탈탄소 전력) |
| **X3** | 자본 흐름 경쟁 | 같은 투자 테마/자금 풀을 두고 경쟁하는 자산은? | BMNR vs 금 ETF (인플레 헤지), CRCL vs PayPal (핀테크 배분) |
| **X4** | 인재 경쟁 | 같은 인재 풀을 두고 경쟁하는 산업은? (인재가 병목인 경우) | AI 스타트업 vs 빅테크 vs 퀀트헤지펀드 (ML 엔지니어) |
| **X5** | 규제 유비 | 다른 산업에서 유사한 규제 궤적을 겪은 사례는? | 크립토 규제 vs 1990년대 인터넷 규제, 담수화 환경규제 vs 원전 인허가 |

**실행 규칙:**
1. L2 Sector Loop에서 **X1, X2** 탐색 의무 — Opportunity Scanner가 이종 경쟁자 제시
2. L3 Value Chain Loop에서 **X2, X4** 탐색 의무 — Moat Analyst가 기능 대체 위협 매핑
3. L4 Company Loop에서 **X1, X3, X5** 탐색 의무 — Company Analyst가 비명시적 경쟁자 식별
4. **이종 비교에서 나온 인사이트는 [THESIS] 태그** — 검증 전 가설
5. 이종 비교가 Edge Test E1(Novelty)를 충족시키는 주요 경로

**산출물:** 각 Loop의 Cross-Comparison 결과를 별도 섹션으로 기록:
```
### 🌐 Cross-Domain Comparison (Loop N)
- 비교 대상: [산업A] vs [산업B]
- 비교 축: X1/X2/X3/X4/X5
- 발견: [인사이트]
- Edge 기여: E1/E2/E3/E4 중 어디에 기여하는지
```

### 8.4 Causal Mechanism Map & Deep Causal Drill 🔬

> 이전 독립 §6(Causal Mechanism Map) + §10.4(Deep Causal Drill)을 통합.
> Deep Loop R1에서 인과사슬을 그리고, Loop마다 한 단계씩 더 파고든다.

**Causal Mechanism Map (R1 의무 — L3/L4):**
```
[투입] → [프로세스] → [핵심 노드(=분석 기업)] → [산출] → [최종 가치]

핵심 질문:
1. 이 인과사슬의 어느 고리가 끊어지면 이 기업이 죽는가?
2. 대안 기술/경로의 현실성과 타임라인은?
3. 이 기업이 인과사슬에서 대체 불가능한 이유는?
```

**Deep Causal Drill (인과 심화 탐구):**

> 1차 병목에서 멈추지 않는다. 5단계까지 인과사슬을 추적한다.

**이전**: Bottleneck Hunter가 1차 병목 식별 → 끝
**이후**: Loop마다 인과사슬을 한 단계씩 더 파고들기

**절차 (Loop 내 R1에서 Bottleneck Hunter 의무):**

```
Loop 1: [메가트렌드] → [1차 병목] 식별
Loop 2: [1차 병목] → [1차 병목의 원인] → [2차 병목] 식별
Loop 3: [2차 병목] → [2차 병목의 원인] → [3차 병목] 식별
...
```

**종료 조건:** "이 단계의 병목을 소유하는 상장 기업이 있는가?" YES이고 시장이 인식하지 못했으면 → Edge Found.

**예시: AI 데이터센터 전력 수요**
```
Loop 1: AI 수요 증가 → GPU 수요 → [1차 병목: 전력 공급]
Loop 2: 전력 공급 부족 → 변압기 부족 → [2차 병목: 변압기 리드타임 3년]
Loop 3: 변압기 리드타임 → 절연유(나프텐계) 부족 → [3차 병목: 특수 원유 정제 능력]
Loop 4: 나프텐 정제 → Nynas AB + Calumet Specialty (비상장/소형) → [4차 병목: 나프텐 원유 매장지 제한]
Loop 5: 나프텐 원유 → 베네수엘라/브라질 해저 → 지정학 리스크
→ Edge: "AI 수요의 5차 파생 병목은 나프텐 원유 지정학" — 시장 인식 제로
```

## 9. 확률 결합 & 포지션 사이징

### Layer별 확률 산출

각 Layer는 독립적으로 **방향성 확률**을 산출한다:

| Layer | 산출물 | 스케일 |
|-------|--------|--------|
| L1 Macro | Regime 확률 (Risk-On / Neutral / Risk-Off) | 각 0-100% (합=100%) |
| L2 Sector | 산업별 매력도 스코어 | 1-10 |
| L3 Value Chain | 밸류체인 내 위치 적합도 | 1-10 |
| L4 Company | 기업 확신도 | 1-10 |
| L5 Chart | 기술적 타이밍 등급 | Strong Buy ~ Strong Sell (1-5) |

### 조건부 가중 곱셈

```
Final Score = w₁·P(Macro) × w₂·P(Sector) × w₃·P(ValueChain) × w₄·P(Company) × w₅·P(Chart)
```

**기본 가중치:**

| Layer | 가중치 | 근거 |
|-------|--------|------|
| Macro | 0.25 | 모든 배가 함께 오르내림 |
| Sector | 0.20 | 산업 선택이 종목 선택보다 중요 |
| Value Chain | 0.10 | 정성적 보완 |
| Company | 0.30 | 개별 기업이 최종 수익 결정 |
| Chart | 0.15 | 타이밍은 중요하나 과도 의존 금지 |

### Regime-Adjusted 가중치

Macro Regime에 따라 가중치 동적 조정:

| Regime | Macro↑ | Company↑ | Chart↑ |
|--------|--------|----------|--------|
| Risk-On (강세) | 0.15 | 0.35 | 0.20 |
| Neutral | 0.25 | 0.30 | 0.15 |
| Risk-Off (약세) | 0.40 | 0.20 | 0.10 |

→ 약세장에서는 매크로가 모든 것을 지배, 강세장에서는 종목 선택이 핵심

### Modified Kelly Criterion

```
Kelly % = (p × b - q) / b
  p = Final Score (승률)
  b = Expected Reward / Expected Risk (보상 배율)
  q = 1 - p

Half-Kelly = Kelly% / 2  (보수적 적용)
Max Position = 30% (단일 종목 상한, 2026-03-24 상향)
Min Threshold = 0.10 (이 이하면 투자 안 함)
```

### 사이징 매트릭스

| Final Score | 확신도 | 포지션 % |
|-------------|--------|---------|
| 0.30+ | ⭐⭐⭐ 최고 | 15-25% |
| 0.20-0.30 | ⭐⭐ 높음 | 8-15% |
| 0.10-0.20 | ⭐ 보통 | 3-8% |
| <0.10 | 낮음 | 0% (관망) |

- **§2 Layer별 전문가 판정 테이블** — 각 전문가의 독립 방향/확신도/Critic 결과
- **§5 Checkpoints** — 30d/90d/180d/1yr 자동 채점 대기

이 구조가 없으면 전문가별 적중률을 측정할 수 없다.
**향후 모든 judgment는 반드시 §2 테이블을 포함해야 한다.**

### 전문가 적중률 DB

`db/expert-accuracy.json` — 주간 전략 리뷰(일요일 cron)가 자동 업데이트:

**측정 지표:**
- **방향 적중률** — PLEDS 방향(Bull/Bear) vs 실제 주가 움직임
- **Critic 효과** — Critic이 abandon시킨 논거가 실제로 틀렸는지 (Critic 정확도)
- **Brier Score** — 확률 예측 정확도 (확신도 vs 실제 결과)
- **Layer 기여도** — 어느 Layer가 최종 판정 변경에 가장 기여했는지

**업데이트 주기:**
- **주간**: 주가 스냅샷 수집 + Kill 체크 (주간 전략 리뷰 cron)
- **30d/90d checkpoint**: 방향 적중 확정 + accuracy 업데이트
- **분기**: Feedback Signal Report → Layer 가중치 조정 검토

**가중치 조정 규칙:**
- 적중률 상위 20% 전문가: 가중치 +10%
- 적중률 하위 20% 전문가: 가중치 -10% + 페르소나 재검토
- Layer 적중률 60% 미만: 해당 Layer 가중치 하향 검토
- **최소 30건 이상 축적 후 조정** (표본 부족 방지)

### Conviction Journal 추적

## 12. Investor Context Layer (ICL) 🧭

> "시스템은 투자자를 이해해야 한다. 동조하기 위해서가 아니라, 조언을 캘리브레이션하기 위해서."

### 14.1 투자자 프로파일

| 항목 | 내용 |
|------|------|
| **총 Arena 자금** | ~$318K (전액 투자용, 생활비/결혼 Vault는 별도 KRW) |
| **연소득** | ~$100K (세후 ~$70K) |

| **레버리지 태도** | 선물 지양 (인지 비용 + 2025 교훈). 주식 내에서 conviction sizing으로 대체 |

### 14.2 ICL 적용 규칙

**A. Conviction Override 존중**
- Shawn이 시스템 권고를 override할 때, 이를 기록하되 존중한다
- Override 사유를 `conviction/` journal에 기록
- 90일 후 복기 — 시스템이 맞았는지, Shawn이 맞았는지 평가
- **시스템은 "틀렸을 수 있는 이유"를 제시하되, 최종 결정을 방해하지 않는다**

**B. Goal-Aligned Advice**
- 모든 포트폴리오 제안 시 ICL 프로파일 참조
- "이 조언이 Shawn의 실제 목표(안전망→사업 자유)에 부합하는가?" 자문
- 단순 수익률 극대화가 아닌, **목표 달성 확률 극대화** 관점

**C. Anti-Sycophancy Anchor + ICL Critic**
- ICL은 동조 도구가 아닌 **캘리브레이션 도구**
- Shawn의 conviction이 데이터와 충돌하면 **솔직하게 지적**
- "Shawn이 원하는 답"이 아니라 "Shawn에게 필요한 답"을 제공
- 지적 후에도 Shawn이 진행하면 → Override 기록 + 존중

**ICL Critic Pass (필수):** 모든 조언/제안 산출 후, 아래 3가지 질문으로 자기 검증:
1. **"이 말이 정말 Shawn에게 도움이 되는가?"** — 편한 말 vs 필요한 말 구분
2. **"Shawn의 인식을 확장하는가?"** — 이미 아는 것을 반복하고 있진 않은지
3. **"팩트에 근거한 합당한 의견인가?"** — 감정/분위기에 편승하지 않았는지

→ 3개 중 하나라도 NO이면 **더 날카롭고 팩트 기반으로 재구성** 후 산출

**D. Trauma-Aware Risk Calibration**
- 2025-10 트라우마로 인한 무의식적 리스크 회피를 인지
- "이 trim/hedge 결정이 데이터 기반인가, 공포 기반인가?" 점검 질문 삽입
- 단, 역으로 과도한 리스크 추구도 견제 — 양방향 캘리브레이션

**E. Dry Powder 전략 인정**
- SGOV/Cash 보유를 "비효율"로 자동 판정하지 않음
- 시장 공포 구간(VIX 35+, -20% 조정) 시 배치 우선순위 제안
- 평시에는 기회비용 알림만 (분기 1회)

### 14.3 "100배 똑똑한 Shawn" 섹션 캘리브레이션

기존 "100배 똑똑한 Shawn이라면?" 섹션에 ICL 반영:

- 단순히 "더 똑똑한 투자자"가 아니라 **Shawn의 실제 상황(연소득, 결혼, 사업 목표)을 아는 더 똑똑한 Shawn**
- "이론적 최적"이 아니라 **"Shawn이 실행 가능한 최선"**을 제안
- 예: "LLY는 좋은 회사지만 당신의 전쟁에 맞는 무기가 아니다" (ICL 맥락)

### 14.4 업데이트 주기

- **ICL은 Shawn이 명시적으로 업데이트를 요청할 때만 변경**
- 연소득, 목표, 리스크 성향 등 주요 변수가 바뀌면 반영
- CRO가 임의로 ICL을 수정하지 않음

### 15.1 변경 시 Cross-Reference Check (의무)

헌법 문서 수정 시 아래 절차를 **기계적으로** 실행:

```
1. 변경 키워드 추출 (예: "10x" → "3x", 전문가 추가/삭제 등)
2. grep -rn "{키워드}" METHODOLOGY.md WORKFLOW.md EXPERTS.md PORTFOLIO.md DATA-SOURCES.md INDEX.md
3. 모든 참조 위치에서 정합성 확인 + 동시 수정
4. 버전 번호 통일 업데이트
5. 변경 사항이 다른 헌법 문서에 영향 미치는지 확인
```
